# 1. 지원 동기

1기 과정에서 여러 프로젝트를 수행하며, 프로젝트 완성도는 구현 능력만이 아니라 초반 요구사항 정의와 팀원 간 공통 이해에서 결정된다는 것을 배웠습니다. 저는 1기에서 겪은 시행착오와 개선 경험을 바탕으로 2기 수강생들이 프로젝트를 안정적으로 완성하도록 돕고, 강사님과 학원의 보조 업무도 책임감 있게 수행하고자 조교에 지원했습니다.

가장 큰 시행착오는 2번째 공장 자동화 프로젝트에서 겪었습니다. 컨베이어 벨트 위 물품을 카메라로 인식하는 기능을 구현할 때, 팀원마다 물품을 먼저 감지한 뒤 촬영할지, 카메라를 계속 동작시키며 인식할지 이해가 달랐습니다. 저는 화이트보드에 물품 감지, 촬영, 이미지 인식, 결과 처리 흐름을 단계별로 정리하며 의견 차이를 구조화했습니다. 그 결과 문제의 핵심이 단순한 방식 선택이 아니라 센서와 카메라의 연동 조건을 명확히 하는 데 있음을 확인했습니다.

이 경험을 통해 프로젝트 초반에는 구현보다 요구사항, 동작 시나리오, 기능 간 연결 구조를 먼저 정리해야 한다는 점을 배웠습니다. 조교로 활동하게 된다면 이러한 노하우를 바탕으로 2기 수강생들이 문제를 구조적으로 파악하고 해결 방향을 찾도록 지원하겠습니다. 또한 강사님과 학원에서 요청하는 문서 업무와 보조 업무의 목적을 정확히 이해하고 성실히 수행하겠습니다.

# 2. 본인의 강점과 약점

저의 강점은 ROS 기반 로봇 제어 기능을 구현할 때, 기능 통합까지 고려해 모듈형 구조로 개발하는 능력입니다. 3번째 자율주행 프로젝트에서 주행 로봇 `pinky_pro`의 네비게이션 기능을 담당하며, 실제 2차선 도로와 유사한 Gazebo 맵에서 도로주행 기능을 구현했습니다. 당시 전체 설계가 충분히 확정되지 않은 상태였기 때문에, 이후 통합 단계에서 패키지 간 의존성과 인터페이스 불일치가 발생할 가능성이 있었습니다.

저는 기존 `pinky_pro` 코드에서 하드웨어 제어와 시뮬레이션에 필요한 부분만 활용하고, 주행 로직은 `araseo_navigation`이라는 별도 패키지로 분리했습니다. 외부 패키지가 알아야 할 인터페이스를 최소화하고 내부 로직은 독립적으로 수정·테스트할 수 있도록 구성한 결과, 통합 단계에서 기능을 안정적으로 연결할 수 있었습니다. 이 경험을 통해 로봇 소프트웨어 개발에서는 단순 구현력뿐 아니라, ROS 패키지 구조와 제어 흐름을 이해하고 통합 가능한 구조로 코드를 작성하는 역량이 중요하다는 것을 배웠습니다.

반면 저의 약점은 프로젝트에서 모호하거나 막히는 부분이 생기면 먼저 혼자 해결하려는 성향입니다. 4번째 피지컬 AI 프로젝트에서 팀장을 맡아 요구사항과 아키텍처를 정리하고, 각 로봇에 명령을 내리며 상태 정보를 받아 전체 흐름을 제어하는 중앙 관제 파트를 담당했습니다. 하지만 로봇별 기능과 UI 연동 방식이 명확하지 않은 상태에서 관제 구조와 API 설계까지 혼자 결정하려 하다 보니 작업량이 과도해졌고, 오히려 구현 방향을 잡는 속도가 늦어졌습니다.

이를 개선하기 위해 강사님의 피드백을 바탕으로 매일 회의를 진행하며 각자 구현 중인 기능, 필요한 입력과 출력, 다른 파트와 연결되어야 하는 지점을 공유했습니다. 그 결과 모호했던 API 구조와 중앙 관제의 역할을 명확히 정리할 수 있었고, 팀 전체의 구현 방향도 안정되었습니다. 이 경험을 통해 팀장은 모든 문제를 혼자 해결하는 사람이 아니라, 문제를 드러내고 역할과 인터페이스를 명확히 하여 팀이 함께 해결하도록 만드는 사람이라는 점을 배웠습니다.

# 3. 위기 극복 경험

저는 최신 모델 적용이 어려운 상황에서 하드웨어 제약에 맞게 구조를 바꿔 모방학습 기반 서빙 기능을 구현한 경험이 있습니다. 프로젝트에서는 로봇팔이 트레이 위 식음료를 집고, 사람이 있는 테이블 방향으로 서빙하는 기능을 구현해야 했습니다. 처음에는 smallVLA와 Groot 같은 최신 VLA 모델을 활용해 픽업과 서빙 동작을 하나의 모델로 학습시키려 했습니다. 그러나 실제 테스트 과정에서 하드웨어 연산 성능이 부족했고, OMX 로봇팔이 해당 모델을 제대로 지원하지 않아 안정적인 구동이 어려웠습니다.

저는 무거운 VLA 모델을 계속 적용하는 것보다 현재 환경에서 실제로 동작 가능한 구조를 만드는 것이 중요하다고 판단했습니다. 그래서 가벼운 ACT 기반 모델로 전환했습니다. 다만 ACT는 비전 기반 동작 복제에는 적합했지만, 픽업과 서빙을 구분하거나 사람이 있는 방향으로 서빙해야 하는 맥락을 스스로 이해하지 못했습니다. 이를 해결하기 위해 픽업 모델과 서빙 모델을 분리하고, 알고리즘이 상황에 따라 필요한 모델을 로드하도록 구성했습니다. 또한 비전 화면에 도형 조건을 입력해 모델이 수행해야 할 동작 범위를 명확히 제한했습니다.

그 결과 제한된 하드웨어 환경에서도 픽업과 서빙 동작을 구현할 수 있었습니다. 이 경험을 통해 로봇 개발에서는 최신 모델을 적용하는 것만큼이나, 실제 하드웨어 제약과 기능 요구사항에 맞춰 적절한 모델을 사용하고 설계하는 판단이 중요하다는 점을 배웠습니다.
